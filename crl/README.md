# `crl/` — Acme-free port of Contrastive RL

A single-process, modern-JAX reimplementation of the training pipeline from the
2022 paper *"Contrastive Learning as Goal-Conditioned RL"* (Eysenbach et al.).
It removes every dead dependency from the original repo — **no Acme, Launchpad,
Reverb, TF, `gym.envs.robotics`, or `mujoco-py`** — so it runs as one process on
Windows/Colab CPU or GPU.

The *algorithm* is a faithful port; only the orchestration is new:

| Original (unreproducible) | Here |
|---|---|
| Launchpad distributed program | `crl/train.py` single-process loop |
| Reverb `EpisodeAdder` + tables | `crl/replay.py` numpy trajectory buffer |
| TF-dataset `flatten_fn` sampler | `crl/replay.py` geometric future-goal sampler |
| `gym.envs.robotics` + `mujoco-py` | `crl/envs.py` (numpy point-env; `gymnasium-robotics` Fetch) |
| Acme `Transition` / `NormalTanhDistribution` | small reimplementations in `losses.py` / `networks.py` |

The critic + all losses (NCE / CPC / C-learning / GCBC, adaptive alpha, the
`random_goals` actor trick) are copied essentially verbatim from
`contrastive/networks.py` and `contrastive/learning.py`.

## Files
- `config.py` — hyperparameters (`Config` dataclass).
- `networks.py` — Haiku critic + tanh-normal policy.
- `losses.py` — loss functions + the JAX update step.
- `replay.py` — trajectory buffer + geometric future-goal relabeling.
- `envs.py` — `point_*` (pure numpy) and `fetch_reach` / `fetch_push` (gymnasium-robotics).
- `train.py` — collect → relabel → SGD → eval loop + CLI.
- `selftest.py` — verifies the sampler distribution and a full learner update.

## Algorithm selection (mirrors `lp_contrastive.py`)
| Method | Flags |
|---|---|
| `contrastive_nce` (default) | *(none)* |
| `contrastive_cpc` | `--use_cpc` |
| `c_learning` | `--use_td --twin_q` |
| `nce+c_learning` | `--use_td --twin_q --add_mc_to_td` |
| `gcbc` | `--use_gcbc` |

## Run locally (no MuJoCo — 2D navigation)
```bash
pip install "jax[cpu]" dm-haiku optax numpy      # already present if selftest runs
python -m crl.selftest
python -m crl.train --env_name point_FourRooms --max_number_of_steps 200000 \
    --min_replay_size 2000 --random_steps 4000
```

## Run on Colab (continuous control — Fetch)
```python
# --- Colab setup cell ---
!pip install -q "jax[cuda12]" dm-haiku optax gymnasium gymnasium-robotics mujoco
import os; os.environ["MUJOCO_GL"] = "egl"     # headless rendering backend

# clone your fork, then:
!python -m crl.train --env_name fetch_reach --use_td --twin_q \
    --max_number_of_steps 1000000 --min_replay_size 10000
```

`fetch_reach` uses goal = gripper xyz (`start_index=0, end_index=3`);
`fetch_push` uses goal = object xyz (`start_index=3, end_index=6`). These are set
automatically in `envs.make_env`.

## Checkpoints, resume, and watching the agent
- **Checkpointing to Google Drive** (`crl/checkpoint.py`): pass `--ckpt_dir <path>`
  (or `Config(ckpt_dir=...)`). Every eval it writes `latest.pkl` (resume),
  `best.pkl` (highest success), and `metrics.json` (full success history) — not
  just once at the end. Add `--resume` to continue from `latest.pkl` after a
  Colab disconnect. A checkpoint is ~3–4 MB (well under any Drive concern).
- **Render a rollout to a GIF** (`crl/visualize.py`) to *see* the env / trained
  policy:
  ```bash
  python -m crl.visualize --env_name fetch_reach --ckpt runs/best.pkl --out reach.gif
  python -m crl.visualize --env_name point_FourRooms --out point.gif   # random policy
  ```
  Fetch frames come from MuJoCo (`render_mode='rgb_array'`, needs `MUJOCO_GL=egl`
  on Colab); point frames are drawn with matplotlib. In a notebook:
  `from IPython.display import Image; Image('reach.gif')`.
- **Colab notebook:** [`notebooks/colab_contrastive_rl.ipynb`](../notebooks/colab_contrastive_rl.ipynb)
  — deps, Drive mount, train, GIF, success-rate plot. All logic stays in `crl/`.

## No dataset to upload
This is **online** contrastive RL: data is generated by interacting with the
simulator during training and stored in the in-memory `TrajectoryBuffer`. There
is no static dataset file to commit or upload to Drive.

## Notes / known differences from the original
- **Throughput, not math, is approximate.** The original coupled actor/learner
  through Reverb's sample-to-insert rate limiter. Here that ratio is the explicit
  `--updates_per_step` knob. The loss computations are identical.
- **Negatives** come from independently sampled trajectories (sampling with
  replacement) rather than the original transpose-shuffle; statistically
  equivalent for the contrastive objective.
- Image-based envs (`*_image`) are wired in the networks (`use_image_obs`) but the
  env wrappers here cover state-based tasks first.
